# AI时代，开发语言应当如何抉择

在正式聊这个话题之前，我们先看几组数据。

<img src="/courses/hi-agent/begin/images/image-1778988637700.png" alt="TIOBE 2026 年 5 月编程语言排行榜" />

按照 TIOBE 2026 年 5 月的编程语言排行榜，Python 排名第 1，C 排名第 2，Java 排名第 3，C++ 排名第 4，C# 排名第 5，JavaScript 排名第 6。TIOBE 同时也特别说明，它统计的是语言热度，不代表哪门语言最好，也不代表哪门语言的代码量最多。换句话说，这个榜单不能直接拿来判断技术选型，但至少能说明一件事：Java 并没有消失，它仍然是全球最主流的编程语言之一。

再看 Stack Overflow 2025 开发者调查。所有受访者中，JavaScript 使用率为 66%，Python 为 57.9%，TypeScript 为 43.6%，Java 为 29.4%；在专业开发者中，JavaScript 为 68.8%，Python 为 54.8%，TypeScript 为 48.8%，Java 为 29.6%。Stack Overflow 还提到，Python 从 2024 到 2025 增长了 7 个百分点，这和 AI、数据科学、后端开发的共同推动有关。

GitHub Octoverse 2025 的信号更有意思。GitHub 的数据里，TypeScript 在 2025 年 8 月第一次超过 Python 和 JavaScript，成为 GitHub 上使用最多的语言；GitHub 认为，这和 AI 辅助开发、类型系统、现代前端框架默认使用 TypeScript 有很大关系。同时，Python 仍然在 AI 和数据科学场景里非常强，GitHub 也明确提到，Python 仍然是 AI 和数据科学工作负载里的主导语言。

所以我们看到的不是一个简单的结论。

- 不是 Java 不行了。
- 也不是 Python 一统天下。
- 更不是 TypeScript 可以包打一切。

更真实的情况是：Java 依然稳稳站在主流语言前排，但 AI 原生生态和开发者工具链的热度，正在明显向 Python 和 TypeScript 倾斜。这也是为什么最近关于“AI 时代该用什么语言”的讨论会越来越多

这是临时起意的一小节，缘起有二：

* 1. 当前我们在做的整个Hi Agent，我计划整体的语言要换到TypeScript了，至于为什么，后续会细说
* 2. 在Spring AI Alibaba社群里，我见证了一场大战，各方吵的不可开交，在讨论，AI时代，Java究竟该何去何从
 
思索良多，我认为这确实是值得一聊的话题，因此在开篇的章节，单独开出一篇来和大家聊聊我对于，AI时代，开发语言的选择问题。


## 这不是语言战争，而是系统分工问题

<Callout type="info">
当然，这篇内容没有任何价值倾向 ，也没有任何语言选择倾向，不是为了吹捧 TypeScript 或 Python。更不是为Java在这个时代挽尊；Just观点分享，放心食用
</Callout>
为什么一开始就写了一篇免责声明，因为，用什么语言这件事情，这种争论太容易吵偏。

一聊 Java，就有人说它稳、成熟、企业级；一聊 Python，就有人说它生态好、AI 原生；一聊 Node，又有人说它适合前后端一体、适合快速做产品。每个说法都有道理，但如果最后只是为了证明某一门语言赢了，那这个讨论就没有太大意义。

因为真实的工程世界，从来不是靠一门语言包打天下的。

尤其到了 AI Agent 这个阶段，一个稍微复杂一点的系统，根本不只是“写一个 Agent”这么简单。它背后还有用户系统、权限系统、组织管理、计费系统、模型网关、工具调用、插件机制、任务调度、日志审计、前端控制台、流式输出、沙箱环境……这些东西放在一起，才是一个完整的 Agent 工程。

我个人更倾向于，“抛开业务需求和团队来聊语言选择，都是耍流氓”；所以在我看来，我们与其争论AI时代，我究竟要用什么语言，还不如来聊清楚：“在一个 AI 系统里，Java 应该负责哪一部分？Python 应该负责哪一部分？TypeScript 又应该负责哪一部分？”，这个角度一换，很多争论，其实就没有那么的尖锐了。

Java 当然有它非常强的地方。比如稳定的后端业务、复杂权限、企业集成、事务一致性、长期维护，这些都是 Java 多年积累下来的优势。你让它去做用户系统、订单系统、审计系统、后台管理，我觉得一点问题都没有，甚至很合适。

但如果我们讨论的是 Agent 的核心编排、工具调用、模型接入、上下文管理、流式交互、插件生态，那 Python 和 TypeScript 确实更顺手。不是因为它们天然高级，而是因为这一波 AI 工具链、框架和社区实践，很多就是从它们那里先长出来的。

所以我越来越觉得，AI 时代的语言选择，不该再带着“阵营感”。

不是说我以前写 Java，所以所有东西都必须 Java；也不是说现在 Python、TypeScript 火，所以就把所有老系统推倒重来。成熟一点的做法，应该是让不同语言站在自己最舒服的位置上。

## 为什么这个问题在 AI 时代突然变尖锐了？

其实 Java、Python、Node 之间的争论，过去也一直存在。

只不过在以前，这个问题没有今天这么刺眼。因为过去大多数业务系统的形态相对明确。一个企业系统，大概率就是用户、权限、订单、审批、报表、接口、数据库这些东西。系统的复杂度主要来自业务规则、数据一致性、并发访问、权限控制和长期维护。那时候的语言选择，更多的是历史遗留的选择，当核心业务从一开始就是Java或者Python构建的，那么在之后的迭代中，大概率是不会轻易切换技术栈的，而且，从实现以及从语言能力的角度来看，大可不必 。

在这个语境下，Java 很容易站得住。Spring 生态成熟，团队好招，工程规范稳定，企业里也有大量现成经验。你要做一个后台系统、管理平台、交易系统、审批系统，选 Java 几乎不会被认为是一个冒险决策。它不一定最性感，但足够稳，足够可靠，也足够让人放心

但 AI 时代来了以后，事情开始变得不一样了。因为我们要构建的系统，不再只是传统意义上的业务系统。尤其是到了 Agent 这个阶段，系统里面突然多了很多以前不常见的东西。

比如模型调用、Prompt 编排、上下文管理、工具调用、RAG、向量数据库、Memory、多 Agent 协作、流式响应、MCP、沙箱执行、任务中断与恢复、人类介入确认、模型评估、运行轨迹回放。

这些东西不是传统 CRUD 系统里自然长出来的。它们更像是一套新的工程问题。

而这套新问题的第一批答案，很多不是从 Java 生态里长出来的。最早的一批模型 SDK、RAG 框架、Agent 框架、评估工具、文档解析工具、向量处理工具，大量集中在 Python 生态里。后来，随着 AI 产品化和前端交互越来越重要，TypeScript 和 Node 又迅速冲了上来。这就导致了一个很现实的现象：

你如果做传统企业后台，Java 依然很舒服。
但你如果做 Agent 编排、工具调用、模型接入、流式交互，Python 和 TypeScript 往往更顺手。

**问题的尖锐，正是从这里来的。**

更麻烦的是，AI 相关的技术变化太快。今天大家还在讨论怎么写一个 ReAct Agent，明天又开始讨论 Graph Agent；今天刚接上某个模型 SDK，过两天又要支持新的多模态接口；今天还在做简单的函数调用，后面又要接 MCP、代码解释器、浏览器操作、文件系统、长期记忆。

在这样一个高速变化的阶段，语言生态的“反应速度”就变得特别重要。

- **Python** 的优势在于，它离模型、数据和实验最近。新论文、新框架、新工具，往往第一时间就会有 Python 版本。你要做 RAG、做评估、做 embedding、做文档处理、做模型实验，Python 的资料、示例和现成轮子都非常多。
- **TypeScript** 的优势则在于，它离产品和交互最近。Agent 应用不是一个黑盒接口，它需要实时展示模型输出，需要展示工具调用过程，需要前端控制台，需要插件市场，需要工作流编排，需要和浏览器、Node 服务、前端状态管理紧密配合。这个时候，TypeScript 的体验就会非常自然。

Java 当然也能做这些，但很多时候不是“能不能做”的问题，而是“做起来顺不顺”的问题。这也是为什么很多 Java 开发者在 AI 时代会突然产生一种失落感。

以前我们熟悉的那套后端优势，好像不再总是站在舞台中央。以前做系统，Java 是默认答案；现在做 AI Agent，很多教程、框架、示例、开源项目，默认答案却变成了 Python 或 TypeScript

于是争论就起来了。

- 有人觉得 Java 依然是企业级系统的最终答案。
- 有人觉得 AI 时代应该全面拥抱 Python。
- 也有人觉得 TypeScript 才是 Agent 产品化的最佳选择。

但在我看来，这种争论之所以激烈，是因为大家讨论的其实不是同一个问题。

- 有人讨论的是企业核心系统。
- 有人讨论的是模型和算法生态。
- 有人讨论的是 Agent 框架和产品交互。
- 有人讨论的是已有系统的维护成本。
- 有人讨论的是新项目的开发效率。

这些东西放在一起吵，当然会吵得不可开交。因为每个人说的都可能对，但只在自己的场景里对。

所以，AI 时代让语言选择变尖锐的真正原因，不是某一门语言突然赢了，也不是某一门语言突然输了，而是软件系统本身变复杂了。

当系统变成这样以后，再用过去那种“我们团队统一用 Java”或者“AI 生态都用 Python”来做判断，就会显得有点粗糙。

语言选择不再是一个单点决策，而变成了架构设计的一部分。

这也是我觉得这个问题值得单独拿出来聊的原因。因为它背后真正讨论的，不是 Java、Python、TypeScript 谁更有面子，而是我们到底要用什么方式，来组织一个 AI 时代的软件工程。

## Java真的掉队了吗？
作为一名Java老兵，其实在过去很长一段时间里，我还是对java抱有很大的热忱的；比如，Hi-Agent原计划还是用Java来做的；我之前很长一段时间也还在做Spring AI相关的教程等等，但是确实，会明显感觉到，Why，为什么要这样做；难道Python不是更快吗？

所以，很多时候，我也会想说：既然 AI 时代的第一波生态都在 Python 和 TypeScript 那边，那 Java 是不是已经掉队了？

仔细思考过后，我觉得这个判断有点太着急了。

Java 确实没有站在这一波 AI 原生生态的最前排。你去看很多 Agent 框架、RAG 教程、模型调用示例、插件协议、工具调用方案，最常见的语言基本就是 Python 和 TypeScript。很多新东西出来以后，第一时间给出的 SDK、Demo 和社区实践，也往往不是 Java。

这一点没必要回避。

如果你是一个 Java 开发者，最近这一两年大概率会有一种感觉：怎么突然之间，热闹好像都不在自己熟悉的地方了？

以前做后端，Java 几乎是默认答案。
现在一聊 Agent，一聊 RAG，一聊工具调用，一聊 MCP，一聊 AI 应用开发，大家的代码示例很可能就是 Python 或 TypeScript。

这种落差是真实存在的。

但这能说明 Java 掉队了吗？

我觉得不能。

因为这里要分清楚两件事：**生态前沿和工程落地**。

Python 和 TypeScript 更靠近生态前沿。它们离模型、实验、工具链、产品交互更近，所以新东西经常先在它们那里冒出来。这个阶段大家追求的是快速试错、快速集成、快速验证，能不能先跑起来很重要。

而 Java 更靠近工程落地。它的强项不是第一时间追热点，而是把一个系统做稳、做久、做大，做成团队能维护、企业能放心、问题能追踪、风险能控制的东西。

这两者不是同一种能力。

所以我更愿意这样说：

<Callout type="info">
Java 不是掉队了，而是它没有成为 AI 原生创新层的默认语言。
</Callout>

这句话听起来有点绕，但很重要。因为 Java 的价值，本来就不应该只用“能不能写 Agent 框架”来衡量。

如果今天我们要做的是一个 AI 聊天 Demo，一个 RAG 实验，一个模型评估脚本，一个 Agent 原型，那我也不会优先选 Java。因为这类东西变化太快，Python 和 TypeScript 确实更顺手。

但如果今天我们要做的是一个真正上线的 AI 产品，它就不只是一个 Agent Demo 了。

它要有用户系统、权限系统、套餐计费、订单支付、企业组织、数据隔离、审计日志、调用限额、API Key 管理、后台管理、风控策略、消息通知、任务记录。再往后走，还会遇到多租户、灰度发布、数据权限、支付对账、异常补偿、成本核算、日志追踪这些一点都不“AI”，但又绕不开的问题。

这些东西当然不是只有 Java 能做。

Python 可以做，Node/TypeScript 也可以做，而且在很多团队里也能做得很好。今天我们已经很难再用“某门语言能不能做企业级系统”来判断技术选型了。只要架构设计合理，工程规范到位，测试、监控、部署和治理体系跟得上，Python 和 Node 一样可以支撑复杂系统

所以问题不是：

> 这些东西谁能做？

而是：

> 在我们的团队和业务场景下，谁做起来更稳、更省心、更容易长期维护？

如果一个团队本来就有成熟的 Java 技术栈，有稳定的 Spring Boot 体系，有现成的用户、权限、计费、审计和后台管理模块，那这些稳定业务层继续用 Java，是很自然的选择。不是因为 Python 或 Node 做不了，而是因为没有必要为了追赶 AI 浪潮，把已经成熟的东西推倒重来。

Java 的价值也正是在这里体现出来的。

它不是唯一答案，但它是很多企业场景里非常稳妥的答案。尤其是面对复杂业务规则、长期维护、大团队协作、历史系统集成和强一致性要求时，Java 过去积累下来的工程经验和生态基础，仍然很有分量。

换句话说，Java 在这里的优势不是“别人做不了”，而是：

> 当一个组织已经围绕 Java 建立了成熟工程体系时，它能用很低的迁移成本，把 AI 能力接进原有业务系统里。

这才是 Java 在 AI 时代真正值得保留的位置

## Python 和 TypeScript 为什么在 Agent 领域更顺手？

说完 Java，我们再来看 Python 和 TypeScript。

这几年，只要聊到 AI 应用开发，尤其是 Agent、RAG、模型评估、工具调用这些方向，Python 和 TypeScript 的存在感都非常强。

这并不是偶然。

它们之所以在 Agent 领域更顺手，不是因为这两门语言天然比 Java 高级，也不是因为 Java 做不了这些事情，而是因为 Agent 这类系统的核心问题，刚好更贴近 Python 和 TypeScript 的优势区间。

先说 Python。

Python 在 AI 领域的优势，几乎不用多解释。模型训练、数据处理、向量检索、文档解析、Embedding、评估实验、各种机器学习和深度学习工具，长期以来都是 Python 生态最活跃。

所以当 AI 应用从“调用一个模型接口”走向 RAG、评估、检索增强、上下文处理、多模态数据处理时，Python 很自然就成了很多人的第一选择。

- 你要处理 PDF、Word、网页、表格、图片 OCR；
- 你要做文本切分、向量化、召回、重排；
- 你要做模型效果评估、自动化测试、数据分析；
- 你要快速试一个新的 RAG 策略，或者验证一个新的 Agent 思路；

这些事情，用 Python 往往就是最省事的。

它的优势不是“写出来一定更优雅”，而是你想要的东西，大概率已经有人写过了。你需要的库、示例、教程、实验代码、社区讨论，基本都能很快找到。

这在一个快速变化的领域里非常重要。

Agent 领域现在还没有完全定型。今天我们讨论的是 ReAct，明天可能变成 Graph Agent；今天还在做简单工具调用，后面又要加长期记忆、任务恢复、人类确认、运行轨迹回放、评估闭环。这个阶段最重要的不是一开始就设计得完美，而是能不能快速试错，快速把想法跑通。

Python 在这方面确实很舒服。

但 Python 也有自己的边界。

它适合做 AI 能力层、数据处理层、实验层，但如果要做一个交互复杂、前后端强绑定、需要大量实时状态同步的 Agent 产品，Python 就不一定是最舒服的那个选择。

这时候 TypeScript 的优势就出来了。

Agent 产品和传统后台系统不太一样。

传统后台系统，很多时候是用户发一个请求，后端处理完，返回一个结果。
但 Agent 产品更像是一条持续发生的事件流。

用户发起任务之后，模型可能先思考，然后调用工具，再读取文件，再执行代码，再返回中间结果，再等待用户确认，最后继续下一步。这个过程不是一次请求响应就结束了，而是一串状态变化。

这就带来了很多前端和交互问题。
- 模型正在输出什么？
- 当前调用了哪个工具？
- 工具执行成功还是失败？
- Agent 现在卡在哪一步？
- 是否需要用户确认？
- 任务能不能暂停？
- 能不能恢复？
- 中间过程要不要展示？
- 运行日志要不要回放？

这些问题看起来是后端问题，但最后一定会反映到产品界面上。

而 TypeScript 最大的优势，就是它同时贴近前端和后端。

前端是 TypeScript，后端也可以是 TypeScript；数据结构可以共享，接口类型可以共享，事件协议可以共享。对于一个 Agent 平台来说，这种统一体验非常有价值。

尤其是 Hi Agent 这类系统，如果未来要做 Agent 调试台、工具调用过程展示、插件管理、工作流编排、流式输出、任务状态追踪，那么 TypeScript 会非常自然。

因为它不是只负责“写接口”，它还可以贯穿整个产品层。

- 前端看到的任务状态，后端也用同一套类型描述；
- 前端展示的工具调用记录，后端也用同一套事件结构推送；
- 插件的输入输出参数，可以直接用 TypeScript 类型约束；
- 模型流式输出、工具调用流、运行日志流，都可以用统一的事件协议组织。

这会明显降低开发时的割裂感。

另外，Agent 领域里还有一个很重要的方向，就是工具和插件。

一个 Agent 系统要真正变强，不能只靠模型本身。它必须能调用外部工具，比如搜索、数据库、文件系统、浏览器、代码执行器、设计工具、办公软件、内部 API 等等。

而 TypeScript/Node 在这方面很适合做“连接层”。

它天生适合处理异步 I/O，适合写 API 网关，适合做 WebSocket、SSE，适合接各种 Web 服务，也适合做插件系统和开发者工具。再加上现在很多前端、Node、MCP 相关生态都在快速发展，TypeScript 就很容易成为 Agent 产品层的主语言。

所以，如果说 Python 更像是 AI 能力的实验室，那么 TypeScript 更像是 Agent 产品的操作台。

Python 适合把模型能力、数据能力、评估能力跑起来。
TypeScript 适合把这些能力组织成一个开发者能用、用户能交互、前端能展示、工具能扩展的产品。

当然，这并不意味着 Python 和 TypeScript 就没有问题。

- Python 的问题在于，项目一旦变大，如果没有类型约束、工程规范和测试体系，很容易从“灵活”变成“随意”。早期写得很爽，后期维护起来可能会痛苦。
- TypeScript 的问题则在于，Node 生态虽然活跃，但依赖复杂、版本变化快，工程治理如果跟不上，也会很容易变乱。尤其是后端业务一旦变复杂，只靠语言本身并不能保证系统稳定。

所以我并不想把 Python 和 TypeScript 描述成 AI 时代的万能答案。

它们只是更适合站在 Agent 领域的前台。

更准确地说：

> Python 的优势在于离模型和数据更近。
>
> TypeScript 的优势在于离产品和交互更近。
>
> Java 的优势在于离稳定业务和长期治理更近。

这么一看，语言之间并不是互相替代的关系，而是各自站在不同位置上。

- 如果一个团队要做 RAG 实验、模型评估、数据处理，用 Python 很合理。
- 如果一个团队要做 Agent 平台、工具调用网关、流式交互、插件系统，用 TypeScript 很合理。
- 如果一个团队要做用户、权限、计费、审计、业务后台，用 Java 也很合理。

真正不合理的，不是选了哪门语言，而是明明系统里有不同类型的问题，却非要用一门语言解决所有问题。

AI 时代的软件工程，恰恰需要我们从这种单语言惯性里走出来。

不是因为过去的语言不行了，而是因为新的系统形态变复杂了。Agent 不是一个函数，也不是一个接口。

它更像一个会持续运行、不断调用工具、不断产生状态、不断和用户互动的工程系统。

在这样的系统里，Python 和 TypeScript 之所以更顺手，不是因为它们赢了语言战争，而是因为它们刚好站在了 Agent 当前最活跃、变化最快、最需要试错的地方。

## 我的结论：语言不是信仰，架构才是答案

写到这里，我其实越来越觉得，AI 时代讨论开发语言，最怕的不是选错语言，而是把语言当成信仰。

一旦把语言当成信仰，讨论就很容易变味。

Java 开发者会下意识维护 Java，觉得它稳定、成熟、企业级，过去那么多大系统都跑在 Java 上，没理由到了 AI 时代就突然不行了。

Python 开发者会觉得，AI 生态本来就在 Python 里，模型、数据、RAG、评估、训练、推理，最活跃的东西都在这里，为什么还要绕一圈？

TypeScript 开发者则会觉得，Agent 产品最终要落到交互、状态、插件、工具调用和前后端协同上，TypeScript 才是最自然的选择。

这些说法都没错。

但如果每个人都只站在自己熟悉的那一边看问题，最后就会变成一句话：

> 我熟悉的，就是最好的。

这显然不是一个成熟的工程判断。

真正的工程判断，应该反过来看。

不是先问我喜欢哪门语言，而是先问这个系统到底要解决什么问题。

- 它是一个快速验证的 Agent 原型，还是一个要长期运行的商业化产品？
- 它更靠近模型和数据，还是更靠近用户和业务？
- 它需要强前端交互，还是需要稳定事务和权限治理？
- 它是从零开始的新项目，还是已经有一套成熟的历史系统？
- 它的核心矛盾是快速试错，还是长期稳定？

这些问题回答清楚了，语言选择自然就没有那么神秘了。

所以我的结论很简单：

> 语言不是信仰，架构才是答案。
>
> AI 时代真正成熟的技术选型，不是选一门“最强语言”，而是让每门语言站在它最该站的位置上。

## TIPS: Hi-Agent为什么转向了TS

其实 Hi-Agent 这次转向 TypeScript，原因没有那么玄乎。

一个很重要的原因是：我希望大家在学习的时候，可以直接在浏览器里运行我提供的代码。

在后面的小节里，我会引入 WebContainer。也就是说，你不需要在本地安装一堆环境，不需要先配 JDK、配 Maven、配 IDE、配各种依赖，也不需要一上来就和环境问题搏斗半天。你可以直接打开页面，在浏览器里编辑代码、运行代码、观察结果。

这件事对于课程来说非常重要。

因为学习 Agent 工程，本来就已经有很多新概念要理解了：模型调用、工具调用、上下文管理、流式输出、插件机制、任务编排。如果一开始还要让大家先花大量时间解决环境配置问题，学习体验就会被打断。

TypeScript 在这里的优势就很明显。

它天然和浏览器、前端工程、在线编辑器、Node 运行时、WebContainer 这些东西贴得更近。对于一个面向学习和实践的项目来说，它能让代码更容易被打开、更容易被修改、更容易被运行，也更容易做成交互式教程。

这并不是说 Java 技术上做不到。

Java 当然也可以做在线运行环境，也可以接沙箱，也可以做远程编译和执行。但问题在于：没有那么方便。

Java 的本地开发体验很成熟，但它天然更依赖完整的后端工程环境。JDK、构建工具、依赖管理、项目结构、启动方式，这些东西对专业开发者来说不是问题，但对学习者来说，很容易变成门槛。

而 Hi-Agent 这门课，我希望它不是一套“你看我讲”的课程，而是一套“你边看边改、边改边跑”的课程。

所以从这个角度看，TypeScript 不只是一个技术选择，也是一个教学体验选择。

我希望大家在学习 Hi-Agent 的时候，能更快进入 Agent 本身，而不是先被环境配置劝退。

这也是 Hi-Agent 转向 TypeScript 的一个非常现实的原因。
